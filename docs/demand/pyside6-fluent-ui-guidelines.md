# PySide6-Fluent-Widgets UI 设计准则

本文档定义一套可迁移到其他 `PySide6-Fluent-Widgets` 工具项目的通用 UI 设计准则。  
它关注的是长期有效的设计原则、分层边界、页面范式、扩展方式和验收标准，而不是某一次整改的实施步骤。

本文档允许引用具体项目中的落地模式作为示例，例如：

- `AppSettings`：现有配置系统与 Fluent 设置卡片之间的桥接方式
- `TaskDescriptor`：任务类页面的统一元数据模型
- `BatchFilesBinding`：UI 与后台执行线程之间的接线层

这些示例用于说明“推荐做法”，不是对其他项目的强制命名要求。

## 1. 文档目标

本准则适用于基于 `PySide6-Fluent-Widgets` 构建的工具型桌面应用，目标是：

- 让 UI 风格统一，而不是每个页面各写一套规则
- 让页面结构在功能增长时依然稳定
- 让表现层与业务层分离，降低后续维护成本
- 让主题、导航、状态反馈和设置体系具备一致性
- 为新增页面、任务、设置项和样式提供稳定扩展方式

本准则优先面向“多页面工具软件”，尤其适合具备以下特征的项目：

- 左侧导航或多页面壳层
- 同时存在配置页、工作台页、文档页
- 需要统一主题和通知机制
- UI 需要持续扩展，但不希望频繁重写骨架

## 2. 核心原则

### 2.1 表现层与业务层分离

UI 层负责：

- 页面布局
- 交互响应
- 状态展示
- 通知分发
- 调用业务接口

UI 层不应承担：

- 文件扫描或数据处理算法
- 参数推导规则
- 配置文件底层读写细节
- 后台执行逻辑本身

推荐保留一层轻量桥接层连接 UI 与业务。例如：

- 用 `TaskDescriptor` 提供任务标题、说明、图标和执行类
- 用 `BatchFilesBinding` 统一线程执行、状态变更与日志转发
- 用 `AppSettings` 负责配置读写，页面只消费映射后的定义

### 2.2 约束优先于局部发挥

页面开发优先级应固定为：

1. 先确认是否已有页面范式或组件模式可复用
2. 再决定是否在统一 QSS 中补规则
3. 最后才考虑页面内极少量动态样式

默认不推荐：

- 在页面里大量写 `setStyleSheet()`
- 对同一类控件使用多套高度、圆角、边距
- 用一次性布局拼接代替可复用骨架
- 为了快速修一个问题，在页面内直接硬编码颜色值

### 2.3 页面类是装配层

一个页面类应尽量薄，只保留：

- 页面初始化
- 布局装配
- 选中态切换
- UI 状态刷新
- 通知或信号转发

如果一个页面类开始同时负责业务推导、配置解析、线程控制和复杂条件判断，通常说明需要继续拆层。

## 3. UI 分层准则

推荐把基于 Fluent 的工具型 UI 分成四层：

### 3.1 应用层

职责：

- 应用启动
- 注册页面
- 注册任务或模块元数据
- 连接页面与业务桥接层

### 3.2 窗口壳层

职责：

- 管理主导航
- 应用主题
- 分发顶层通知
- 挂载一级页面

推荐使用 `FluentWindow` 作为壳层，而不是自行维护多套 `QMainWindow + Dock + StackedWidget` 组合。

### 3.3 页面层

职责：

- 组织某一类用户任务的布局和交互
- 按统一模式呈现内容
- 不直接承担底层业务实现

### 3.4 视觉系统层

职责：

- 主题入口
- 主题模式与有效主题的映射
- 应用级 QSS
- 颜色与布局 token
- 主题相关图标的刷新规则

## 4. 主题与视觉系统准则

### 4.1 主题模式与有效主题分离

主题设计必须区分两个概念：

- 主题模式：用户保存的值，例如 `Light`、`Dark`、`Auto`
- 有效主题：当前实际生效的明暗主题，最终只会是 `Light` 或 `Dark`

推荐规则：

- 持久化保存用户的主题模式，而不是保存解析后的有效主题
- `Auto` 应允许在运行时跟随系统主题变化
- 当系统主题检测失败时，回退到 `Light`

### 4.2 统一主题入口

主题应用应集中到单一入口模块，统一负责：

- 解析主题模式
- 应用 `setTheme()`
- 应用主题色 `setThemeColor()`
- 按有效主题加载应用级 QSS
- 在 `Auto` 模式下响应系统主题变化

不要让各页面自行判断主题、各自加载样式或各自维护明暗状态。

### 4.3 视觉 token

建议定义一组项目级视觉 token，至少包括：

- 主强调色
- 成功色
- 警告色
- 错误色
- 暗色背景
- 暗色面板
- 暗色卡片
- 主文本色
- 次级文本色

这组 token 不一定要一步到位做成完整设计系统，但必须保证应用级样式使用的颜色有统一来源。

### 4.4 布局 token

建议至少固定以下布局常量：

- 页面外边距
- 卡片内边距
- 区块间距
- 行内控件间距
- 控件目标高度
- 卡片圆角

新增页面时优先沿用这些常量，不要每页重新发明一套尺寸。

### 4.5 应用级 QSS 规则

QSS 应主要用于应用级规则，例如：

- 标题层级
- 次级说明文案
- 卡片背景与边框
- 列表 hover / selected 态
- 日志区与文档区样式
- 状态文案颜色

推荐规则：

- 明暗主题规则成对维护
- 全局改动优先回到统一 QSS
- 页面内只保留必须动态变化的局部样式

### 4.6 主题相关图标

长期显示的图标必须跟随有效主题刷新，而不是在初始化时固定成一次性的浅色或深色版本。

通常包括：

- 一级导航图标
- 工作台侧栏图标
- 设置侧栏图标
- 分类图标

如果某类图标在 Light 或 Dark 中可见性不稳定，应优先修正统一图标使用方式，而不是单独替换某一个页面的图标资源。

## 5. 页面架构范式

以下三类页面范式适合多数工具型桌面应用，也适合作为可复用模板。

### 5.1 任务中心范式

适用于“左侧选择任务，右侧执行和查看详情”的工作台页面。

推荐结构：

1. 页头
   - 标题
   - 说明文案
   - 全局上下文摘要
2. 左侧任务列表
   - 任务标题
   - 图标
   - 选中态
   - 必要的运行态标记
3. 右侧任务工作区
   - 概览区
   - 操作按钮
   - 分段详情区

推荐原则：

- 全局上下文只保留一份，不要在每个任务卡片里重复维护
- 任务列表只负责选择与状态概览，详细说明放到右侧
- 执行、状态、日志位置保持稳定
- 每个任务的日志应独立维护，切换任务不丢日志

推荐元数据模型：

- 一个统一的任务描述结构，例如 `TaskDescriptor`
- 字段至少覆盖：唯一 key、标题、说明、图标、执行类、默认参数

### 5.2 设置中心范式

适用于“统一配置入口 + 分类/分组展示”的页面。

推荐结构：

1. 页头
   - 标题
   - 说明文案
2. 一级切换
   - 如 `通用设置 / 任务设置`
3. 子页骨架
   - 左侧选择栏
   - 右侧详情区

推荐原则：

- 设置页应有清晰的信息架构，不要把所有分类一次性纵向堆满
- 左侧栏负责分类或实体选择，右侧负责配置详情
- 右侧详情区优先使用 `ScrollArea + SettingCardGroup`
- 保存策略应在项目级统一，不建议不同分类使用不同保存逻辑

配置桥接建议：

- 如果项目已存在稳定配置系统，不必为了 Fluent 外观强行迁移到 `QConfig`
- 可以用轻量适配层把现有配置映射为卡片定义
- `AppSettings` 这类对象适合作为“配置读取 + 写回 + 变更信号”的桥接层示例

设置项映射建议：

- 布尔值：`SwitchSettingCard`
- 有预定义选项的值：组合框型卡片
- 普通文本：输入框型卡片
- 密钥或敏感字段：密码输入型卡片

### 5.3 帮助 / 文档页范式

适用于“文档源稳定，但阅读体验需要统一”的页面。

推荐结构：

1. 页头
2. 顶部分段切换
3. 文档阅读卡片

推荐原则：

- 继续复用已有 Markdown 文档源
- 不在帮助页内引入复杂文档系统
- 用 `SegmentedWidget + stacked widget` 组织多份文档
- 外部链接统一交给系统浏览器打开

## 6. 组件选型与复用准则

推荐优先使用 Fluent 原生组件处理以下场景：

- 一级导航
- 卡片容器
- 设置卡片
- 分段切换
- 通知
- 列表与选择器

若原生组件与现有项目配置体系不直接兼容，可采用“轻量包装”而不是整套重写。例如：

- 继承 `SettingCard` 包装输入框和组合框
- 保留现有配置系统，只桥接展示层

判断某个组件是否应抽象的标准：

- 是否会在多个页面重复出现
- 是否有稳定的视觉和交互规则
- 是否已经存在可复用元数据模型

## 7. 状态、通知与日志准则

工具型应用通常应明确区分三种反馈：

### 7.1 全局即时提示

适用于：

- 阻断性错误
- 成功完成提示
- 需要用户立即注意的短时信息

推荐统一经由顶层窗口壳分发，例如 `InfoBar.success / error / warning / info`。

### 7.2 页面内状态文本

适用于：

- 当前工作目录摘要
- 选择状态
- 当前任务状态
- 持续存在的说明信息

这类信息不应依赖瞬时弹窗。

### 7.3 日志流

适用于：

- 执行过程消息
- 逐步输出
- 任务级运行记录

不要把所有过程消息都升级成全局通知，也不要让所有结果都只留在日志里。通知、状态文本和日志必须职责清晰。

## 8. 扩展准则

### 8.1 新增任务

推荐流程：

1. 准备业务处理类
2. 新增任务元数据
3. 用统一注册流程接入页面和执行桥接层
4. 补充必要默认参数
5. 验证任务列表、描述、日志和运行态

推荐让任务元数据成为 UI 与执行逻辑之间的唯一真相来源。

### 8.2 新增设置项

推荐流程：

1. 先在配置源中增加字段
2. 再在配置桥接层中增加映射
3. 验证是否可自动生成卡片
4. 仅在输入类型不覆盖时新增新的卡片适配组件

不要让页面直接依赖底层配置文件路径或 JSON 细节。

### 8.3 新增一级页面

推荐流程：

1. 新建页面类
2. 设置稳定的 `objectName`
3. 在窗口壳层统一注册
4. 接入页面自己的通知信号

不建议绕开主导航体系，直接散落多个独立窗口承载一级页面职责。

### 8.4 新增应用级样式

推荐流程：

1. 先为组件定义对象名或属性
2. 再在明暗两套 QSS 中成对补规则
3. 最后评估是否应回收为更正式的 token 或通用组件规则

## 9. 禁止项与反模式

以下做法应视为高风险或不推荐：

- 在页面里继续大量堆 `setStyleSheet()`
- 为每个设置项单独拼一套布局
- 在主窗口之外再维护另一套页面切换逻辑
- 让页面直接读取底层配置文件并解析 JSON
- 让页面类同时承担 UI、业务推导、线程执行和配置持久化
- 只修单个页面的颜色问题，而不回到统一主题规则
- 在多页面项目里缺少统一通知入口

## 10. UI 验收基线

任何基于本准则的 UI 迭代，至少应满足以下验收规则：

### 10.1 启动与导航

- 应用可正常启动
- 主窗口壳可正常显示
- 一级导航切换正常
- 默认首页行为明确且稳定

### 10.2 主题一致性

- `Light / Dark / Auto` 行为正确
- `Auto` 在运行中可跟随系统主题变化
- 系统主题检测失败时有明确回退策略
- 应用级 QSS 与主题相关图标会同步刷新

### 10.3 可读性与对比度

- 所有长期显示的交互控件在 Light / Dark 下都清晰可辨
- 至少检查：
  - 卡片背景
  - 输入框
  - 下拉框收起态
  - 下拉弹层
  - 开关
  - 占位文本
  - 禁用态
  - 滚动条
  - 焦点态
  - 侧栏图标

### 10.4 布局稳定性

- 在 `1280x720` 下无主要遮挡、重叠、跑版
- 窗口缩放时主导航、侧栏、详情区、日志区可正常伸缩
- 长文本、长路径、长日志不会破坏布局

### 10.5 行为回归

- 设置项显示与保存行为正确
- 页面切换不丢关键状态
- 任务执行相关页面的日志、运行态和通知行为符合预期
- 帮助或文档页的分段切换与外链行为正常

## 11. 已知限制与取舍

### 11.1 Fluent 风格的视觉取舍

`PySide6-Fluent-Widgets` 更接近 Fluent / WinUI 语言，而不是各平台原生外观。

这意味着：

- 它更容易建立统一的跨平台工具界面
- 但不以平台原生视觉为第一目标

这应被视为一种主动设计取舍，而不是缺陷。

### 11.2 无界面环境兼容性

部分 `FluentWindow` 或 `qframelesswindow` 相关行为在 `QT_QPA_PLATFORM=offscreen` 下可能存在限制。

推荐测试策略：

- 页面类可做离屏实例化验证
- 真实窗口壳验证应在桌面环境中执行

### 11.3 配置桥接模式的代价

保留现有配置系统并桥接到 Fluent 设置卡片，优点是迁移风险低；代价是某些依赖 `QConfig` 的高级卡片能力需要自己适配。

这是合理取舍，只要项目边界清晰。

## 12. 项目内示例映射

在 RD-tools 里，这套准则当前可以对应到以下示例：

- `AppSettings`
  - 现有配置系统与设置页卡片的桥接层
- `TaskDescriptor`
  - 任务元数据中心
- `BatchFilesBinding`
  - UI 与后台执行线程之间的信号桥接层
- `ui/theme.py`
  - 统一主题入口
- `ui/qss/light/app.qss` 与 `ui/qss/dark/app.qss`
  - 应用级明暗样式

其他项目不必复用这些名字，但应复用它们所代表的职责分离方式。
