# RD-tools 懒加载设计文档

本文档描述  当前采用的懒加载设计，目标是把“软件启动速度”和“单个任务首次执行速度”拆开治理，而不是用一套过度复杂的插件系统一次性解决所有问题。

当前设计的核心不是“所有东西都尽量晚加载”，而是分两层处理：

1. 通用的模块级懒加载
2. 选择性的重任务依赖级懒加载

这两层叠加后，启动阶段只保留轻量的 UI、配置和调度逻辑；真正的任务实现和重依赖只在需要时才进入运行态。

## 1. 设计目标

本次懒加载设计解决的是两个不同层面的问题：

- 软件打开时，不应该因为少数重任务模块而统一变慢
- 某些重任务第一次执行时，不应该把大量三方依赖提前压进启动链

因此设计目标是：

- 启动时不导入具体任务类
- 设置页、任务页和帮助页可以在任务模块未导入的情况下正常工作
- 点击执行某个任务时，才加载它对应的实现模块
- 对 `ECGHandler`、`GenSubtitles` 这类重任务，再进一步把重量级依赖下沉到真正用到时
- 不引入 DI 容器、插件系统、运行时单例或独立 worker 进程

本设计不解决：

- 打包产物的 `PyInstaller --onefile` 启动开销
- 所有任务首次点击时的绝对零等待
- 模型缓存、跨任务共享 backend、进程隔离等更重的架构问题

## 2. 总体思路

当前采用的是“两层懒加载”：

### 2.1 第一层：模块级懒加载

这是全体任务统一采用的基础机制。

含义是：

- 启动时只注册任务元数据
- 不直接导入 `modules.xxx` 里的具体任务类
- 用户点击执行任务后，才按 `module_path + class_name` 动态加载对应类

这一层解决的是“软件打开慢”的主问题。

### 2.2 第二层：依赖级懒加载

这是只对部分重任务启用的增强机制。

含义是：

- 即使任务模块本身已经延后到执行时才导入
- 模块内部仍然可能有 `matplotlib`、`faster_whisper`、`torch` 之类的重量级依赖
- 对这些重依赖，继续从模块顶层下沉到真正使用的函数内部

这一层解决的是“某个重任务第一次执行时的局部卡顿”。

## 3. 当前架构

当前懒加载链路如下：

```mermaid
flowchart LR
    A["应用启动"] --> B["build_task_descriptors() 只构建元数据"]
    B --> C["FileWindow 注册任务入口"]
    C --> D["用户点击执行任务"]
    D --> E["确认弹窗 + 参数展示"]
    E --> F["创建 one-shot BatchFilesBinding"]
    F --> G["TaskLoader.load_class(module_path, class_name)"]
    G --> H["动态导入对应任务模块"]
    H --> I["实例化 handler"]
    I --> J["设置工作目录并执行任务"]
```

当前这条链上，真正加载任务模块的入口是：

- `main.py`
  - `launch_batch_operation()`
  - `BatchFilesBinding.run()`
- `core/task_loader.py`
  - `TaskLoader.load_class()`

## 4. 第一层设计：模块级懒加载

### 4.1 核心原则

模块级懒加载的关键原则只有一句：

- 启动时只认识“任务是谁”
- 执行时才认识“任务类是什么”

因此 `TaskDescriptor` 不再持有类对象，而只保存元数据：

- `key`
- `title`
- `description`
- `icon`
- `module_path`
- `class_name`
- `settings_group`
- `default_params`

这样带来的效果是：

- 任务页可以正常展示任务列表
- 设置页可以正常展示任务设置入口
- 任务确认弹窗可以展示当前设置
- 但任务实现模块在这一阶段仍未导入

### 4.2 统一加载器

为避免在 UI 层到处手写 `import modules.xxx`，当前通过统一加载器负责动态导入：

- `TaskLoader.load_class(module_path, class_name)`

当前 `TaskLoader` 的职责很单一：

- 通过 `importlib.import_module()` 加载模块
- 通过 `getattr()` 解析类对象
- 按 `(module_path, class_name)` 缓存类对象
- 使用锁保护缓存访问，保证并发调用时的线程安全

它不负责：

- 缓存实例
- 管理任务生命周期
- 管理设置

### 4.3 执行时机

模块级懒加载的触发时机不是“切换到任务页”，也不是“打开设置页”，而是：

1. 用户点击执行
2. 通过目录校验
3. 通过确认弹窗
4. 启动本次执行线程
5. 在线程内调用 `TaskLoader.load_class()`

因此当前行为是：

- 软件启动时，不会统一导入所有任务模块
- 某个任务第一次执行时，才会加载该任务模块
- 同一进程内再次执行该任务时，通常会复用 Python 模块缓存和 `TaskLoader` 的类缓存

### 4.4 为什么要改成 one-shot binding

模块级懒加载落地后，原先“每任务一个常驻 binding + 常驻 handler”的结构就不再合适。

当前改成了：

- 每次执行新建一个 `BatchFilesBinding`
- 每次执行新建一个 handler 实例
- 执行结束后释放本次 binding

这样做的原因是：

- 启动时不必为了注册任务而提前构造 handler
- 当前任务参数在每次执行时重新读取
- 线程边界更清晰，不需要让 UI 长期持有运行态对象

## 5. 第二层设计：选择性的依赖级懒加载

### 5.1 为什么不是所有任务都做

模块级懒加载已经解决了“应用打开慢”的主问题。

因此第二层不是默认对所有任务一视同仁，而是按收益决定：

- 某任务模块很重，才继续做依赖级懒加载
- 某任务模块已经足够轻，就保持简单实现

这一步的目标是优化重点任务，而不是把代码整体复杂度无差别抬高。

### 5.2 当前已纳入第二层的任务

目前明确进入“依赖级懒加载”范围的任务有三个：

1. `ECGHandler`
   - 延后加载 `numpy`
   - 延后加载 `pandas`
   - 延后加载 `matplotlib.pyplot`
   - 延后加载 `scipy.fft`
   - 延后加载 `scipy.signal`

2. `GenSubtitles`
   - 延后加载 `faster_whisper.WhisperModel`
   - 保留按需尝试 `torch`
   - 避免在模块导入阶段直接拉起 `faster_whisper` 依赖链

3. `TwistImgs`
   - 延后加载 `cv2`
   - 继续保留 `numpy` 在模块内的常规导入
   - 目标是把 OpenCV 从任务模块导入路径继续下沉到真正执行图片变换时

## 6. 当前任务分层策略

可以把当前任务分成三类：

### 6.1 A 类：模块级懒加载即可

这类任务模块本身不轻，但首次执行成本仍可接受，不值得继续增加实现复杂度。

当前大致包括：

- `DicomToImage`
- `MergeColors`
- `SplitColors`
- `BiliVideos`
- `FilesRenamer`
- `MacPoopScooper`

### 6.2 B 类：模块级 + 依赖级都需要

这类任务的依赖链已经足够重，模块级懒加载只能把等待从“启动时”挪到“首次执行时”，还不够。

当前明确属于这类的：

- `ECGHandler`
- `GenSubtitles`
- `TwistImgs`

### 6.3 C 类：基础设施层，不纳入懒加载

这类模块必须在启动时可用，因为它们属于调度和配置框架的一部分。

包括：

- `modules.app_settings`
- `modules.files_basic`
- `core.task_loader`
- `ui.task_descriptor`

## 7. 设置与懒加载的关系

懒加载设计要成立，设置系统必须和任务类对象解耦。

当前做法是：

- 设置页不再依赖 `operation_cls.__name__`
- 统一使用 descriptor 中的 `settings_group`

这样带来的好处是：

- 设置页可以在任务模块未导入时正常工作
- 配置结构继续沿用旧类名分组
- 不需要为了读设置而提前导入任务类

当前正式语义是：

- 修改设置后立即写回配置文件
- 不再尝试把设置实时推送给已创建或正在运行的任务实例
- 新设置在下次执行该任务时生效

## 8. 构造阶段消息补齐

懒加载后有一个新问题：

- handler 是在线程里动态创建的
- 某些任务会在 `__init__()` 里发告警或抛异常

如果不补一层过渡机制，构造阶段消息可能会丢。

因此当前增加了 bootstrap reporter：

- `BatchFilesBinding` 在实例化 handler 前安装 reporter
- `FilesBasic.send_message()` 检测到 reporter 时，除了保留原有 pending 逻辑，还会立即把消息转发到页面任务日志
- handler 构造成功后，再连接正式 `result_signal`
- 构造完成后移除 bootstrap reporter

这样可以保留：

- 参数修正告警
- 初始化提示
- 构造失败前的上下文信息

需要明确的是，这套补齐机制只覆盖通过 `FilesBasic.send_message()` 发出的构造阶段消息；如果某个任务在构造阶段直接 `print`、走其他日志通道，或在第一次 `send_message()` 之前就抛异常，这部分信息仍然不会自动进入页面任务日志。

## 9. 线程与对象边界

当前懒加载方案对线程边界有明确要求：

- `TaskLoader` 可以跨线程缓存类对象
- handler 实例绝不跨线程共享
- UI 不再长期持有 handler 引用
- 不允许为了方便再把 handler 拉回 UI 线程做实时属性读写

这意味着：

- descriptor 是 UI 层的稳定元数据
- handler 是执行层的一次性运行对象
- 两者职责必须分开


## 10. 当前设计的边界

### 10.1 它不是插件系统

虽然 descriptor 里已经有 `module_path` 和 `class_name`，但当前仍然是仓库内置任务，不是外部插件发现机制。

### 10.2 它不是运行时共享缓存框架

例如：

- `faster_whisper` 模型复用
- 跨任务共享模型实例
- 共享 `matplotlib` backend 状态

这些都不在当前设计范围内。

### 10.3 它不保证所有任务首次点击都零等待

懒加载只是把等待移到“真正需要时”，不是消灭所有等待。

如果某个任务本身就依赖很重，它第一次执行仍然可能有初始化开销。
