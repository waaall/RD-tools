# UI 后续整改规格：设置中心重构、主题联动与 Auto 主题

这是一个旧文档，不代表现在设计实现。

## 摘要
- 继续保留顶层导航三页：`任务中心`、`设置`、`帮助`。
- `设置` 页改为内部双子页结构：顶部 `SegmentedWidget` 切换 `通用设置` 和 `任务设置`。
- 两个子页统一采用“左侧选择栏 + 右侧设置详情”的骨架，只复用信息架构和视觉样式，不强求复用同一个 widget 类。
- `theme` 配置扩展为 `Light | Dark | Auto`，保存值固定为 `"Auto"`，系统主题检测失败回退 `Light`。
- 所有需要长期显示的主题相关图标都改为跟随“有效主题”实时刷新，不只修任务列表。

## 关键实现变更
### 1. 主题系统
- 将 `Display.Apparence.theme` 的可选值从 `["Light", "Dark"]` 扩展为 `["Light", "Dark", "Auto"]`，不改 JSON 路径和字段名。
- 主题解析与应用逻辑必须区分：
  - “主题模式”是用户保存值：`Light`、`Dark`、`Auto`
  - “有效主题”是当前实际生效值：`Light` 或 `Dark`
- 顶层主题入口改为接受并保留 `Auto`，调用 `QFluentWidgets` 的 `Theme.AUTO`，不要再把非 `Light` 一律降成 `Dark`。
- 接入 `SystemThemeListener` 到窗口壳层：
  - 应用启动时初始化一次。
  - 当设置值为 `Auto` 时，监听系统主题变化并立即重刷应用主题。
  - 当设置值为 `Light` 或 `Dark` 时，停止监听或忽略监听事件。
- 应用级 QSS 仍按“有效主题”加载 `light/app.qss` 或 `dark/app.qss`。
- Auto 模式下系统主题切换时，不需要重启应用；设置页、任务页、帮助页都必须立即更新。

### 2. 设置页信息架构
- `设置` 页顶部增加一级切换：`通用设置`、`任务设置`，默认进入 `通用设置`。
- `通用设置`：
  - 左侧栏固定显示 `General`、`Network`、`Display`。
  - 顺序与当前 `AppSettings.get_main_categories()` 中这三类一致。
  - 右侧只渲染当前选中分类的设置项，不再把所有分类一次性纵向展开。
- `任务设置`：
  - 左侧栏来源固定为 `TaskDescriptor`，使用任务页同源的标题、顺序、图标。
  - 过滤规则固定为：仅显示在 `Batch_Files` 下存在对应设置组的任务。
  - 匹配规则固定为：用 `TaskDescriptor.operation_cls.__name__` 对应 `Batch_Files` 二级 key。
  - 当前仓库下会显示有设置项的任务，不显示无设置项任务；无设置项任务不做占位。
- 两个子页共用同一套页面骨架：
  - 左侧选择栏放在卡片容器中。
  - 右侧详情区为 `ScrollArea + SettingCardGroup`。
  - 保存策略继续保持即时保存，失败才弹错误提示。
- UI 展示名把 `Batch Files` 改为 `任务设置`，但内部 `Batch_Files`、`settings.json` 结构、`AppSettings` 属性命名全部保持不变。

### 3. 设置项组织与卡片规则
- 通用设置右侧仍沿用现有映射规则：
  - 布尔值 `SwitchSettingCard`
  - 枚举值 `ComboBoxSettingCard`
  - 普通文本 `LineEditSettingCard`
  - `api_key` 继续密码输入
- 通用设置右侧的卡片分组规则：
  - 选中 `General` 时，直接渲染该分类全部项。
  - 选中 `Network`、`Display` 时，按当前 path 中分类后的子段落分组，例如 `Serial`、`Internet`、`Apparence`、`Motion`。
- 任务设置右侧的卡片分组规则：
  - 默认按所选任务渲染一组设置卡片。
  - 如果未来某任务 path 在任务类名之后还有更深层级，则这些更深层级再映射成多个 `SettingCardGroup`。
- 默认选中规则：
  - 首次进入设置页时默认 `通用设置 -> General`。
  - 切换到 `任务设置` 时默认选中左侧第一个有设置项的任务。
  - 本次不要求记忆上次子页或上次侧栏选择。

### 4. 图标与深浅主题可读性
- 以下图标统一要求跟随有效主题变化：
  - 任务中心左侧任务列表图标
  - 设置页左侧任务设置列表图标
  - 设置页内部分类图标
  - 顶层导航中必要的页面图标
- 主题联动图标应统一使用可随主题同步更新的图标接口，不允许在初始化时固化一次浅色或深色版本。
- 设置页的视觉验收标准固定为：
  - 所有交互控件在 `Light` / `Dark` 下都必须可读可辨。
  - 范围至少包括：卡片背景、输入框、下拉框收起态、下拉弹层、开关、占位文本、禁用态、滚动条、焦点态。
- 这次整改以全局样式和统一组件行为为主，不接受只对单个控件临时补丁式修色。

## 测试计划
- 主题选项测试：
  - `theme=Light`、`Dark`、`Auto` 都能正确显示、保存、重启后保持。
  - `Auto` 写回配置文件时保存的是 `"Auto"`，不是解析后的值。
  - 系统主题检测失败时，应用回退 `Light`。
- 运行时主题切换测试：
  - 手动在设置中从 `Light/Dark` 切到 `Auto`，无需重启即可进入跟随模式。
  - Auto 模式下系统主题变化时，应用级 QSS、设置页控件和主题相关图标立即刷新。
  - 非 Auto 模式下系统主题变化不得影响应用当前主题。
- 设置页结构测试：
  - 设置页顶部可切换 `通用设置` / `任务设置`。
  - `通用设置` 左侧仅显示 `General`、`Network`、`Display`，右侧只显示当前分类内容。
  - `任务设置` 左侧只显示有设置项的任务，顺序、标题、图标与任务页一致。
  - 当前无设置项的任务不出现在 `任务设置` 左侧。
- 可读性测试：
  - Light/Dark 下所有设置交互控件都具备足够对比度。
  - 任务列表、设置侧栏、设置分类、必要导航图标在两种主题下都清晰可见。
- 回归测试：
  - 即时保存逻辑不变。
  - `Batch_Files` 内部路径和 `AppSettings.changed_signal` 传播链路不变。
  - 任务执行、帮助页、现有顶层导航行为不回退。

## 默认约定
- `TaskDescriptor` 是任务设置侧栏的唯一 UI 元数据真相；`Batch_Files` 只决定“该任务是否有设置项”以及具体设置内容。
- 任务设置侧栏的过滤条件以 `operation_cls.__name__` 命中 `Batch_Files` 二级组名为准。
- 本次不引入中文化的 `General / Network / Display` 标签，不增加搜索、收藏、拖拽排序或记忆上次选中状态。
- 本次不改配置结构、不改 `Batch_Files` 内部 key、不做迁移脚本。
